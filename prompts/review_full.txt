Ты — система глубокого ревью кода и архитектуры. Проведи исчерпывающую проверку проекта.

## 1. GRACE Integrity

### Semantic Markup
- Каждый файл в src/ имеет: FILE, VERSION, MODULE_CONTRACT, MODULE_MAP, CHANGE_SUMMARY?
- Каждая функция имеет START_CONTRACT / END_CONTRACT с PURPOSE, INPUTS, OUTPUTS, SIDE_EFFECTS?
- Все START_BLOCK / END_BLOCK: парные, уникальные имена, ~500 токенов?
- Нет ли блоков без пары (осиротевших START без END)?

### Knowledge Graph (docs/knowledge-graph.xml)
- Каждый модуль из src/ представлен как <M-xxx>?
- Все аннотации (fn-, type-, const-, class-) соответствуют реальным экспортам в коде?
- CrossLink-и актуальны? Нет ссылок на удалённые/переименованные символы?
- Нет ли модулей в коде, отсутствующих в графе?

### Development Plan (docs/development-plan.xml)
- Все export-* реализованы в коде?
- Контракты (inputs/outputs/errors) совпадают с реальными сигнатурами?
- Статусы фаз актуальны?

### Requirements (docs/requirements.xml)
- Каждый UC-* покрыт кодом? Укажи непокрытые.
- AcceptanceCriteria выполнимы с текущим кодом?

## 2. Code Quality

### Корректность
- Есть ли баги, race conditions, необработанные исключения?
- Async код: правильно ли используются await, asyncio, event loops?
- Threading в GUI: все ли обновления UI через root.after()?
- Файловые операции: encoding указан? Пути через Path?

### Типизация
- Все ли сигнатуры функций имеют type hints?
- Нет ли невалидных аннотаций (callable вместо Callable, и т.д.)?
- Dataclass-ы: все поля типизированы?

### Безопасность
- API-ключи: не хардкодятся? Не попадут в git?
- .gitignore покрывает: .env, config.json, __pycache__, *.pyc?
- Нет ли SQL injection, command injection, path traversal?
- Пользовательский ввод валидируется?

### Обработка ошибок
- Сетевые ошибки (таймаут, 429, 5xx) обрабатываются с retry?
- Невалидный JSON от LLM: retry + graceful degradation?
- Отсутствие файлов: информативные сообщения?
- GUI не зависает при ошибках?

### Производительность
- Есть ли блокирующие вызовы в async коде?
- Файлы читаются целиком в память — допустимо для ожидаемых размеров?
- Incremental flush работает корректно?
- Нет ли утечек ресурсов (незакрытые файлы, клиенты)?

## 3. Архитектура

### Связность модулей
- Есть ли циклические зависимости?
- Нарушается ли направление зависимостей (LAYER N зависит от LAYER N+1)?
- Можно ли заменить один модуль без изменения остальных?

### Data Flow
- Данные проходят: Parser -> Batcher -> Extractor -> Writer -> Excel?
- Нет ли мест, где данные теряются или дублируются?
- Промпт передаётся от GUI до JSON/Excel метаданных без потерь?

### Расширяемость
- Легко ли добавить новую модель? Новый промпт? Новый формат вывода?
- Конфигурация гибкая или захардкожена?

## 4. Тесты

- Покрытие: все ли модули имеют тесты?
- Edge cases: пустой ввод, невалидный JSON, огромные файлы?
- Моки: API-вызовы замоканы? Тесты не зависят от сети?
- Изоляция: тесты не влияют друг на друга? Используют tmp_path?

## 5. Упаковка и доставка

- requirements.txt актуален?
- PyInstaller: собирается? .exe запускается? Пути корректны в frozen mode?
- .env и prompts/ работают рядом с .exe?

## Формат отчёта

Для каждой находки:
```
[SEVERITY] FILE:LINE — Описание проблемы
  Рекомендация: ...
```

Severity: CRITICAL (ломает работу), WARNING (потенциальная проблема), INFO (улучшение).

В конце: таблица покрытия UC-* и сводка по категориям.
